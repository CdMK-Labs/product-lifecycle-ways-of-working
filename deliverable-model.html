Deliverable Model

What this page is for

This page shows what to document across the product lifecycle. The purpose is to support decisions, delivery and live product management using a small set of reusable deliverables, rather than many disconnected documents.

The model stays aligned with the financial model, governance processes and our fundamentals. It is not about adding documentation for its own sake. It is about helping teams create the right level of structure for the stage they are in.

Deliverable packs

01 Early Shaping

Opportunity Brief

Frames the problem, the expected value and the investment rationale. Used to decide whether to move into discovery.

 02 Early Shaping

Discovery Pack

Captures what was learned in discovery: problem understanding, options explored, risks and a recommendation on next steps.

 03 Development

Business Case & Delivery Decision Pack

The main approval deliverable. Brings together the value case, cost case and delivery plan for commitment and sign-off.

 04 Execution

Delivery Pack

Tracks scope, progress, quality and dependencies throughout build and delivery. Updated continuously as work progresses.

 05 Execution

Release & Readiness Pack

Confirms the change is ready for go-live: release readiness, operational setup and adoption preparation.

 06 Operations

Product Review Pack

The recurring live-product review covering service health, usage and adoption signals, and improvement decisions.

Templates and minimum viable starting points are available on each deliverable pack page.

How to use this model

Start with the lifecycle stage you are currently in. Use the relevant deliverable pack as the minimum documentation needed to support that stage.

  • Do not create more material than the stage requires
  • Update existing deliverables rather than creating new ones
  • Link to source tools, dashboards or working notes instead of duplicating content
  • Use the stage exit criteria to define what is good enough

Principles

  • Document to support decisions, delivery and operations, not for its own sake
  • Keep deliverables as light as possible for the stage
  • Treat deliverables as information objects, not fixed file types
  • Use one source of truth where possible
  • Keep the lifecycle model and financial model aligned
